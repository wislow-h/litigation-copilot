# Upstage 제품 사용 리뷰 — Litigation Copilot 구축·비교 검증

## 1. 테스트 개요 — 왜 이걸 만들었나

최근 지인이 진행 중인 소송을 변호사와 함께 대응하게 됐는데, **소송기록이 300~1,000페이지에 달해** 당사자가 직접 다 읽어내기가 현실적으로 불가능했다. 게다가 소송기록은

- 분량이 방대하고,
- 법률 용어가 대부분이라 비전문가가 내용을 이해하기 어렵고,
- 설령 다 읽더라도 "그래서 지금 무엇을 해야 하는지"를 전문가 도움 없이 계획하기 어렵다.

Upstage 제품군이 특히 **문서(Document) 처리에 특화**되어 있다는 점에서, 이 문제에 실질적인 도움을 줄 수 있겠다고 보고 직접 제품을 만들어 검증해 보았다.

## 2. 설계 방식

### 2.1 사용 제품
- **Document AI (Document Parse)** — 소송기록(스캔본·HWP 포함)의 파싱/OCR
- **Solar Pro 3** — 추출·요약·추론(타임라인/액션 제안)

### 2.2 비교군
성능 수준을 가늠하기 위해, 가장 범용적으로 쓰이는 **OpenAI 제품**과 동일 과제를 나란히 비교했다.

### 2.3 단계별 사용 API와 선택 이유

| 단계 | Upstage | OpenAI | 선택 이유 |
|---|---|---|---|
| **Parse(파싱·OCR)** | [**Document Parse**](https://console.upstage.ai/docs/capabilities/document-parse) `/v1/document-digitization` (OCR auto, async 최대 1,000p, Markdown 출력) | 전용 파싱 API 없음 → [**Files API**](https://platform.openai.com/docs/api-reference/files) + [Responses `input_file`](https://platform.openai.com/docs/guides/pdf-files) (멀티모달이 직접 판독, ~100p마다 분할) | 스캔본·HWP·표가 많은 소송기록은 레이아웃 보존 파싱이 핵심. OpenAI는 별도 파싱 산출물이 없어 LLM 비전 판독으로 대체 |
| **Analyse(추출·타임라인)** | [**Solar Pro 3**](https://console.upstage.ai/docs/capabilities/chat) + [json_schema(strict)](https://console.upstage.ai/docs/capabilities/chat/structured-outputs) | [**gpt-5.5**](https://platform.openai.com/docs/models) + [Structured Outputs](https://platform.openai.com/docs/guides/structured-outputs) | 단순 필드 추출을 넘어 이벤트 분류·기한 계산·요약까지 필요 → 추론 LLM. 출력 스키마를 강제해 공정 비교 |
| **Instruct(액션 제안)** | [**Solar Pro 3**](https://console.upstage.ai/docs/capabilities/chat) + json_schema(strict) | [**gpt-5.5**](https://platform.openai.com/docs/models) + Structured Outputs | 현 단계에 맞는 행동 옵션을 장단점·시급도와 함께 생성 |

- 공정 비교를 위해 **두 API에 동일 프롬프트·동일 출력 스키마**를 사용해 변수를 "API/모델 차이"로만 좁혔다.
- Analyse와 Instruct는 같은 모델(Solar Pro 3 / gpt-5.5)을 쓴다. 표에서 단계를 나눈 건 *역할*(타임라인 추출 vs 액션 제안) 구분일 뿐 모델 버전 차이가 아니며, 구현상 두 출력은 하나의 구조화 출력 스키마로 함께 생성된다.
- `json_schema(strict)` / Structured Outputs는 외부 라이브러리가 아니라 **API에 내장된 기능**으로, 요청 시 JSON Schema를 넘기면 모델 출력이 그 스키마에 맞게 강제(constrained decoding)된다. 양사에 같은 스키마를 넘겨 출력 형태를 공통화했기에 같은 UI에 렌더하고 공정 비교가 가능했다.

### 2.4 테스트 조건
- 파일 형식: **PDF · DOCX · HWP**
- 언어: **한글 · 영문** 문서
- 분량: **10페이지 ~ 324페이지** 등 다양
- 문서 내 **이미지·표·그림 등 비정형 데이터 다수 포함**
- 양 API에 **동일 프롬프트 + 동일 출력 스키마** 적용

## 3. 프로젝트 설명

**주요 기능 & 플로우** :

> 소송기록 업로드 → 사용할 모델 선택(Upstage / OpenAI / 둘 다) → 업로드 및 분석 실행 → **사건 타임라인 추출** → **액션 아이템을 시급도·장단점 중심으로 제시**

- 대용량 스트리밍 업로드(최대 200MB), 두 파이프라인 병렬 실행, 단계별 진행상태·처리시간 실시간 표시
- 결과는 좌우 2단으로 Upstage vs OpenAI를 나란히 비교

## 4. 결과

도입 / Parse / Analyse / Instruct / 비용, 5개 파트로 정리한다.

### 4.1 도입 — 쉬움
OpenAI SDK 호환 덕분에 `base_url`만 바꿔 기존 코드를 그대로 붙였다. 키 발급부터 첫 호출까지 마찰이 거의 없었다.

### 4.2 Parse — Upstage의 명확한 우위 (특히 한국 문서)
- **OpenAI는 HWP 파일을 기본적으로 처리하지 못함** — 별도 변환/부가 설정이 필요.
- **Upstage Document Parse는 HWP를 그대로, 빠르게 처리.** 스캔본 OCR + 표/제목 레이아웃을 보존한 Markdown을 돌려준다.
- 한국·엔터프라이즈 문서에서 범용 멀티모달 LLM엔 없는 명확한 차별점이다.

### 4.3 Analyse — 타임라인 추출

#### 4.3.1 완성도는 문서 분량·내용에 따라 갈림
- **짧은(100페이지 미만) 문서:** Upstage가 간결하고 핵심적으로 타임라인을 뽑음.
- **300페이지 이상(물류회사 배임 소송 건):** OpenAI가 훨씬 보기 좋게 요약함. 이벤트를 29개로 압축적으로 정리한 반면, Upstage는 169개로 매우 granular(중복·노이즈 포함)했다.
- 즉 "정답"이 갈리기보다, 문서 규모에 따라 강점이 뒤바뀌는 양상이다.

#### 4.3.2 응답 지연·안정성

**현상(데이터).** 324페이지 스캔본 추출에서 지연은 **입력 크기가 아니라 출력 토큰 수에 비례**했다(약 100–120 tok/s). 그리고 글자 한 자 안 바꾼 **동일 입력**을 반복 호출했을 때 응답이 **21s ~ 1,086s로 요동(편차 51.7배), 6회 중 4회 hang**. 성공 시 출력 토큰은 2,605~2,789로 일정 → 모델이 하는 "작업"은 같고 **서빙 지연만 변동**한다.

| 회차 | 1 | 2 | 3 | 4 | 5 | 6 |
|---|---|---|---|---|---|---|
| 결과 | ❌ 180s | ✅ 23.0s | ❌ 1086.6s | ✅ 21.0s | ❌ 180s | ❌ 979.1s |

(최소 21.0s · 최대 1,086.6s · 평균 411.6s · N=6. 소규모 표본이므로 더 큰 표본(N=30+)과 출력량 대조군으로의 추가 검증 여지가 있다.)

**hang이 반복된 조사 과정.**
1. **최초 실행:** 324p 스캔본을 양 파이프라인에 동시 투입. OpenAI는 약 16분에 완료(이벤트 29개), Upstage는 `Connection error`로 실패. 로그상 파싱(OCR)은 성공했고, 실패 지점은 그다음 Solar LLM 호출이었다.
2. **재실행:** 견고성을 위해 재시도·타임아웃을 넣고 다시 돌렸더니, 이번엔 첫 청크(1/6)에서 8분 넘게 멈춤. 응답도 에러도 없이 매달려 진행이 안 됨.
3. **크기 가설 검증:** 동일 규모(약 41K 토큰, 5만 자)의 합성 입력으로 Solar를 호출 → 4.4초에 성공. → 입력 크기 자체는 원인이 아님.
4. **내용 가설 검증:** 실제로 멈췄던 바로 그 청크를 단독으로 다시 호출 → 8.9초에 성공. 12K·6K로 줄여도 모두 빠르게 성공. → 특정 내용 때문도 아니고 일시적(transient) 멈춤으로 좁혀짐.
5. **전체 14청크 순차 측정:** 한 번은 0개 멈춤으로 전부 성공, 단 소요는 출력 토큰에 비례해 4초~72초로 분산(이벤트 밀집 구간일수록 큰 JSON 생성 → 느림).
6. **동일 입력 6회 반복:** 위 표처럼 4/6 hang, 21s~1,086s. → 입력/내용이 아니라 **API 서빙단의 latency 불안정성**임을 확정.
7. **부수 현상:** 멈춘 호출이 클라이언트 타임아웃(180s)을 넘겨 ~1,000s까지 매달림(연결이 끊기지 않고 응답만 지연) → 클라이언트 타임아웃만으로는 깔끔히 회수되지 않았다.


### 4.4 Instruct — 액션 제안: 품질은 대등, "관점"이 다름
- 두 모델 모두 현 단계에 맞는 행동 옵션을 제시하며 품질 차이는 거의 느끼기 어렵다.
- 다만 맥락이 약간 다르다.
  - **Upstage(Solar):** 가능한 옵션을 최대한 많이, 중립적으로 나열하는 경향.
  - **OpenAI(gpt-5.5):** 피고인 입장에서 피해를 줄이는 쪽으로 옵션을 좁혀 제시하는 경향.
- 용도에 따라 호불호가 갈릴 수 있다(폭넓은 탐색 vs 입장 맞춤).

### 4.5 비용 — 표면 금액은 OpenAI가 저렴, 그러나 구조를 봐야 함

**Upstage 실제 사용량 :**

| 항목 | 수량 | 비용 |
|---|---|---|
| **Document Parse (Standard)** | 2,003 페이지 | **$20.03** |
| Solar Pro 3 — Input | 596,786 토큰 | $0.08 |
| Solar Pro 3 — Output | 120,411 토큰 | $0.07 |
| Solar Pro 3 — Cached Input | 49,840 토큰 | $0.00 |
| **합계** | | **$20.19** |

**OpenAI 실제 사용량 :**

| 항목 | 수량 | 비용 |
|---|---|---|
| Responses & Chat Completions | 15 requests / 367,485 tokens | — |
| **합계(Total Spend)** | | **$4.05** |

| 비교 | Upstage | OpenAI |
|---|---|---|
| 과금 방식 | 파싱 = **per-page**, LLM = per-token | 전부 **per-token**(파싱이 토큰에 흡수) |
| 이번 테스트 총액 | **$20.19** | **$4.05** |
| 비용 지배 요인 | **OCR(페이지)가 99%** ($20.03/$20.19) | 입력/출력 토큰 |

**핵심 인사이트**
언뜻 보면 Upstage($20.19)가 OpenAI($4.05)보다 5배 비싸 비용 경쟁력이 없어 보인다. 하지만 구조를 뜯어보면 결론이 달라진다.

- **비용의 99%는 Document Parse(per-page)** 이고, LLM(Solar)은 $0.16에 불과했다. 비용을 좌우하는 건 LLM이 아니라 **OCR 페이지 수**다.
- 그 2,003페이지는 핵심 테스트 문서(324p) 한 건을 재시도·재실행하며 OCR을 반복해 누적된 것이다 → **캐싱/멱등 설계만으로 대부분 제거 가능**(실제로 OCR 결과 캐시를 붙여 재분석 시 페이지 재과금을 없앴다). 한 번만 파싱하면 324p ≈ $3.24 수준.
- 게다가 Upstage가 제공하는 가치는 단순 텍스트화가 아니다 — **한국어 특화, HWP 포맷 지원, 표·이미지 등 비정형 데이터의 안정적 해석.** 이런 문서를 OpenAI로 처리하려면 별도 변환·전처리 비용과 품질 리스크가 든다.
- **결론:** 캐싱 등 운영 전략으로 per-page 비용을 통제할 수 있고, 한국 문서 처리 품질까지 고려하면 **국내 시장에서는 충분히 경쟁력 있는 비용**이라고 본다.
- 덧붙여, **운영 리스크가 곧 비용**이라는 점도 드러났다 — §4.3.2의 hang·재시도가 늘면 per-page OCR 재과금이 함께 늘어난다. 즉 안정성은 단위경제(unit economics) 이슈이기도 하다.

## 5. 마무리

**보완하면 좋을 점**
- **대용량 구조화 출력의 응답 안정성** — tail latency·hang 비율을 줄이는 서빙 안정화 + 구조화 출력 스트리밍.
- **느린/멈춘 응답에 대한 클라이언트 가이드 문서화** — 아래 검토 참고.
- **비용 관측·통제 도구** — per-page 모델용 캐싱/멱등 가이드, 사용량 알림.

**Upstage 문서 검토 — "클라이언트 계약"이 실제로 빠져 있나.**
- 문서화돼 있음: 레이트리밋(예: Solar Pro 3 100 RPM / 50,000 TPM), 429/5xx에 대한 지수백오프 재시도 예제, 스트리밍(`stream=True`, SSE), 에러 코드 표. (출처: [Rate limits](https://console.upstage.ai/docs/guides/rate-limits), [Error codes](https://console.upstage.ai/docs/resources/error-codes), [API Reference](https://console.upstage.ai/api/docs/for-agents/raw))
- 빠져 있음(이번 hang과 직결): 권장 클라이언트 타임아웃 값, 요청 최대 처리 시간(max duration), 느리게/멈춘 응답에 대한 처리 가이드. 동기 호출의 취소(cancel) 시맨틱은 언급이 없고, async 작업 취소는 "API로 지원하지 않음"으로 명시돼 있다.
- 즉 "재시도(429)"는 안내하지만, 이번에 겪은 **"200도 429도 아닌 채 수십 분 멈추는 응답"** 에 대한 계약(언제 끊고/취소할지)이 없다. 이 부분 보강이 가장 실용적이다.

**그럼에도 강점은 분명하다.** 한국·엔터프라이즈 문서(HWP·스캔본·표) 처리는 범용 멀티모달 LLM이 따라오기 어려운 실질적 차별점이고, OpenAI SDK 호환으로 도입 마찰이 거의 없으며, per-page 과금은 캐싱과 결합하면 대용량 배치 비용을 예측 가능하고 통제 가능하게 한다. 위 안정성·클라이언트 가이드 영역이 보강되면 "데모는 좋은데 프로덕션은 불안"이라는 마지막 의구심까지 사라질 것이라고 본다. 다만 한국향 문서 Parse에만 강점이 있다면 시장 한계가 분명할 것이라.. 글로벌 경쟁력은 다른 측면에서도 개발이 되어야 할 것이라는 생각이 든다.
